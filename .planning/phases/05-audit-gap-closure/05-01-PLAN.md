  <files>web/src/lib/chat/api.ts</files>
  <action>
Add a fetchSessionLogs function to api.ts that calls GET /api/sessions/:id/logs?limit=200 and returns the raw log array.

Add a logsToMessages function that transforms raw SSE logs into Message[] suitable for ChatStore.messages. The logs endpoint returns objects shaped like `{ ts: number, type: string, ...eventPayload }`. The transformation logic:

1. Iterate through logs in order.
2. For `type === 'user_message'`: create a Message with role 'user', content from the `text` field, timestamp from `ts`, isStreaming false.
3. For `type === 'done'`: create a Message with role 'assistant', content from the `response` field, timestamp from `ts`, isStreaming false. Attach any pending tool calls collected since the last done event.
4. For `type === 'tool'`: accumulate into a pending ToolCall[] buffer (same shape as ChatStore does during streaming). Filter out `find_tool` events. Set status to hadError ? 'error' : 'done', durationMs to 0.
5. For `type === 'error'`: create a Message with role 'error', content from the `error` field.
6. For synapse events (synapse_start, synapse_question, synapse_answer, task_start, task_done, tasks_complete): build SynapseState incrementally and emit as a role 'synapse' message. Follow the same state machine logic as ChatStore.onEvent — qa phase for questions, executing phase for tasks, complete phase on tasks_complete or done.

Each Message gets a deterministic id via `crypto.randomUUID()` (fine for hydrated history — these are display-only, not server-tracked).

Export both functions. Return type for fetchSessionLogs: `Promise<Record<string, unknown>[]>`. Return type for logsToMessages: `Message[]`.

Handle fetch errors: fetchSessionLogs should return an empty array on any failure (never throws) so that session switch still works even if the logs endpoint is unavailable — the user just sees an empty chat as before.
